# Ship Use Cases First

## Decision
Before investing in more architecture or infrastructure work, **build and ship demo-able use cases**.

## Why
- Miles (non-technical founder) needs to see tangible results
- Abstract planning doesn't drive product-market fit
- Demos > documentation for validation and fundraising
- Quick wins build momentum and investor confidence

## How to Apply
1. **Identify use cases** that solve real user problems
2. **Build MVPs** with minimal scope (1-2 week sprints)
3. **Ship to production** (even if rough)
4. **Get feedback** from users
5. **Iterate** based on real data, not theory

## Current Priority Misalignment
- Past: Heavy focus on v5 architecture (pools, supervisor, etc.)
- Correct: Build 2-3 working products, let architecture follow

## Examples of Good Use Cases
- EdgeBoard betting dashboard (MVP done, needs shipping)
- Sports +EV betting recommendations (proven model + live odds)
- Prediction market bond trading (high-probability contracts)
- NotebookLM-powered research automation

## Next Architecture Decision
Wait until use cases prove what patterns are actually needed.
Avoid over-engineering for hypothetical scale.
